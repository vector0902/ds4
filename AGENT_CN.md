# Agent 笔记

`ds4.c` 是专为 DeepSeek V4 Flash 设计的推理引擎，不是通用的 GGUF 加载器。目标是构建一个小型、易读、高性能的 C 代码库，仅在 Metal 必需时使用 Objective-C，Metal 内核位于 `metal/` 目录下。

## 目标

- 保持整模型 Metal 图推理作为主要生产路径。
- 始终确保 SSD 流式传输、CUDA、分布式推理、Metal 默认推理等其他部分的修复不会影响彼此。
- 保持模型加载使用 mmap 支持（Metal 默认情况）；不要预读取整个 GGUF 文件。SSD 流式传输中专家加载需明确设计：分配缓冲区，从磁盘快速读取，始终尝试通过在进行共享专家和已加载路由专家推理时预加载缺失的路由专家来隐藏加载延迟。在 SSD 流式预填充模式下，始终尝试使用当前层的推理时间来加载下一层以隐藏加载开销。
- CPU 后端保持纯 CPU，仅作为参考/调试代码使用。
- 正确性优先于速度。不要保留存在注意力、KV 缓存或 logits 偏差的高速路径。
- 通过实时 KV 复用和磁盘 KV 检查点实现本地长对话会话。

## 质量规则

- 保持实现简洁、锋利、易于理解。尝试写出优雅优美的代码。不要满足于第一个想到的方案，要寻找最极简且更优的设计。不要引入混乱：极其脆弱的仅修复特定情况的补丁式代码、死代码、无用代码以及过度复杂的代码。
- 在模型机制、缓存生命周期、内存策略或 API 编排不明显的地方为重要推理代码添加注释。
- 优先在实现旁添加注释，而非单独的设计文档。
- 保持注释简洁且富有信息量：解释为何选择特定形状、顺序、缓存边界或内存策略。
- 保持公共 API 精简。CLI/服务器代码不应了解张量内部细节。
- 不要通过标志永久引入语义变体。当验证唯一发布路径时，诊断开关是可以接受的。
- 不要引入 C++。

## 安全

- 避免在 macOS 上进行大型 CPU 推理运行；CPU 路径之前曾在大映射时暴露内核 VM 故障。
- 不要并发运行多个超大模型进程。实例锁是有意为之。

## 布局

- `ds4.c`：模型加载、分词器、CPU 参考代码、Metal 图调度、会话、磁盘缓存有效负载序列化。
- `ds4_cli.c`：命令行交互、linenoise REPL、交互式对话处理。
- `ds4_server.c`：OpenAI/Anthropic 兼容 HTTP API、工作队列、流式传输、工具调用映射、磁盘 KV 缓存策略。
- `ds4_metal.m`：Objective-C Metal 运行时和内核封装。
- `metal/*.metal`：计算内核。
- `tests/`：单元测试和实时集成测试。
- `misc/`：被忽略的笔记、实验和旧规划材料。

此列表不完整，请查看其他文件以获取更多信息。

## 测试

使用 `make` 进行构建验证。当有模型和 Metal 可用时，使用 `make test` 进行单元/回归测试。仅在有意测试 API 表面时使用实时服务器测试。

在以下各项可能受影响的主要变更时，必须确保：

1. 测试正常 Metal 路径，并确保速度仍保持在之前水平。
2. 测试 SSD 流式传输路径。
3. 如果分布式推理可能受影响，则进行测试，但需先向用户确认。
4. 检查更改是否可能破坏 CUDA，并请求用户授权访问 CUDA 机器以实际测试一切是否正常。
